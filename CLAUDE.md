# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

**PLINDER** (Protein-Ligand INteractions Dataset and Evaluation Resource) is a Python
package + data pipeline for building, distributing, and evaluating a large annotated
dataset of protein-ligand docking systems. The repo produces two things:

1. The **dataset** itself (parquet collections + structure archives), generated by
   `plinder.data` and published to the public GCS bucket `gs://plinder`.
2. The **`plinder` Python package** (published to PyPI) that downloads and interacts
   with that dataset, plus an evaluation harness for scoring docking predictions.

The dataset is versioned independently from the code via two controls:
- `PLINDER_RELEASE` — month stamp of the last RCSB sync (default `2024-06`)
- `PLINDER_ITERATION` — iterative development within a release (default `v2`)

The package version itself is derived from git tags via `setuptools_scm`.

## Package layout (`src/plinder/`)

The package is a namespace split into four subpackages with very different dependency
and usage profiles:

- **`plinder.core`** — the *consumer-facing* API. Pure-pip-installable. Manages config,
  lazily downloads/syncs dataset assets from GCS to a local cache, and provides query
  and structure-loading APIs. This is what end users `import`.
- **`plinder.data`** — the *dataset generation* pipeline. Heavy non-pip dependencies
  (foldseek, mmseqs2, openstructure, plip, reduce — Conda/Docker only). Most users
  never touch this.
- **`plinder.eval`** — the docking *evaluation harness*. Depends on `openstructure` for
  metrics. Exposes the `plinder_eval`, `plinder_stratify`, `plinder_plot` CLIs.
- **`plinder.methods`** — currently a placeholder.

### `plinder.core` key entry points

Top-level re-exports (`from plinder.core import ...`): `get_config`, `get_plindex`,
`get_manifest`, `get_split`, `PlinderSystem`.

- `plinder.core.index.system.PlinderSystem` — object wrapping a single system by ID
  (e.g. `1a3b__1__1.B__1.D`); lazily fetches that system's files.
- `plinder.core.scores` — the query API over the parquet collections. Prefer these over
  reading parquet directly: `query_index`, `query_protein_similarity`,
  `query_ligand_similarity`, `cross_protein_similarity`, `cross_ligand_similarity`,
  `query_clusters`, `query_links`. **duckdb** is the preferred reader engine; the
  internal query layer (`scores/query.py`) translates pyarrow-style filters into SQL.
- `plinder.core.loader.dataset.PlinderDataset` — a `torch.utils.data.Dataset` built
  from a split; requires the optional `loader` extra (`torch`).
- `plinder.core.utils.config` — OmegaConf-based config. **Note:** `data.plinder_dir` is
  always derived from `plinder_mount` + `plinder_release`/`plinder_iteration`; never set
  it manually.

### `plinder.data` pipeline architecture

The end-to-end dataset build is an implicit DAG. `pipeline/pipeline.py` (`IngestPipeline`)
is a thin interface between config and the actual business logic in `pipeline/tasks.py`.

The DAG is expressed by naming convention, not a framework: a *stage* is a method
`{stage}` on the pipeline, optionally paired with `scatter_{stage}` (chunks inputs for
parallel/distributed execution) and `join_{stage}` (reduces outputs). `run_stage()`
wires these together; the canonical order lives in `tasks.STAGES`:

```
download_rcsb_files → download_alternative_datasets → make_dbs → make_entries →
structure_qc → make_system_archives → make_ligands → compute_ligand_fingerprints →
make_ligand_scores → make_sub_dbs → run_batch_searches → make_batch_scores →
collate_partitions → make_mmp_index → make_components_and_communities → make_splits →
compute_ligand_leakage → compute_protein_leakage → make_links → make_linked_structures →
score_linked_structures
```

This structure mirrors an external orchestration framework (e.g. metaflow); `run()` is
a convenient way to run the whole DAG locally. Raw-data ingestion lives in
`pipeline/io.py`. See `docs/contribution/pipeline.md` for what each stage reads/writes.

Generated parquet schemas (protein similarity, ligand tanimoto, clusters, splits, links)
are defined in `plinder.data.schemas` — see the schema section of
`docs/contribution/pipeline.md`.

## Development

`plinder.core` is pip-installable, but `plinder.data`/`plinder.eval` need non-pip
dependencies, so development uses **Conda + Docker**.

```console
$ mamba env create -f environment.yml   # foldseek, mmseqs2, openstructure, plip, reduce
$ mamba activate plinder
$ pip install -r requirements_data.txt  # GitHub-only deps (incl. Linux torch)
$ pip install -e ".[dev]"               # dev = lint + test + type extras
$ pre-commit install
```

Linux is fully supported; non-Linux users should use the Docker path
(`docker-compose.yml` + `dockerfiles/`).

## Common commands

Testing/typing/linting go through **tox** (mirrors CI):

```console
$ tox -e py310-test   # runs tests inside Docker via flows/docker.py
$ tox -e py310-type   # mypy src (strict)
$ tox -e py310-lint   # pre-commit on all files (ruff format + ruff check)
```

The `test` tox env actually shells out to `python flows/docker.py test --dirty` — it
builds/runs the test image and mounts the working tree. Tests run with
`PLINDER_OFFLINE=true` and `PLINDER_LOG_LEVEL=10`.

To run pytest directly (inside an activated Conda env or the container):

```console
$ python -m pytest tests/core/test_core_scores.py            # one file
$ python -m pytest tests/core/test_core_scores.py::test_name # one test
$ python -m pytest -n auto                                   # parallel (pytest-xdist)
```

`pytest.ini` enables `--doctest-modules` and `--doctest-glob='*md'`, so **docstrings and
markdown code blocks are executed as tests** (including the schema/example snippets in
docs). It also enforces `--cov-fail-under=60`.

The `flows/docker.py` helper drives the Docker lifecycle directly:

```console
$ python flows/docker.py build           # build the app image
$ python flows/docker.py test --dirty    # run tests, mounting current tree
$ python flows/docker.py run --it        # interactive shell in the container
```

### Running the data pipeline

```console
$ python flows/data_ingest.py            # or: python -m plinder.data.pipeline.pipeline
```

Pipeline runs are parameterized by YAML configs under `flows/configs/v2/` (per-stage
configs like `make_entries_ligands.yaml`, `make_splits.yaml`) and split definitions
under `flows/configs/split_configs/`.

## Conventions & gotchas

- **Code style:** ruff (line length 88, only `F`+`I` rules enabled — formatting via
  `ruff format`, not enforced E/W). Relative imports are banned (`ban-relative-imports`).
  mypy runs in `strict` mode on `src`.
- **Set `PLINDER_OFFLINE=true`** to skip MD5 checksum comparison between local cache and
  GCS — `plinder.core` does this comparison on every asset access by default, which is
  slow/network-dependent. Tests rely on this being set.
- **`PLINDER_LOG_LEVEL=10`** for debug logging (uses Python logging levels).
- The dataset is **lazily downloaded on access** — querying assets that aren't cached
  locally will trigger GCS downloads unless offline.
- Known data caveats (see README): use `query_index` for corrected `entry_release_date`;
  nucleic-acid receptor complexes may not save correctly; `ligand_binding_affinity`
  queries are disabled due to a BindingDB parsing bug.

## Evaluation harness CLIs

- `plinder_eval` (`eval/docking/write_scores.py`) — score docking predictions. Input is
  a `predictions.csv` (cols: `id`, `reference_system_id`, `receptor_file`, `rank`,
  `confidence`, `ligand_file`) plus a `split.parquet`. Writes per-pose JSON + a summary
  `scores.parquet`. Computes lDDT-PLI, bisy_rmsd, lddt_lp (and protein/DockQ/QS metrics
  with flags).
- `plinder_stratify` (`eval/docking/stratify_test_set.py`) — labels test systems by max
  similarity to train and stratifies into novelty classes (`novel_pocket_pli`,
  `novel_ligand`, `novel_protein`, `novel_all`, `not_novel`).
- `plinder_plot` (`eval/docking/make_plots.py`) — generate leaderboard plots.

See `docs/evaluation.md` for full metric definitions and example output.
